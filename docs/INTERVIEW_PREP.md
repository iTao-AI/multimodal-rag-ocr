## ✨ 项目亮点

### 亮点 1: 重排序模块 (Technical Depth)

**问题**: 向量检索召回的文档相关性不够精确

**解决方案**:
```
检索流程:
1. 向量召回 top-50 (快速)
2. Reranker 精排 top-5 (精确)
3. 送入 LLM 生成

准确率：85% → 92% (+7%)
```

**技术细节**:
- 使用交叉编码器 (Cross-Encoder)
- 召回阶段：双塔模型 (快速)
- 重排序阶段：精细交互 (精确)

**面试话术**:
> "我发现单纯的向量检索在某些场景下相关性不够，特别是 query 和文档长度差异大时。所以我添加了重排序模块，先用向量快速召回 50 篇文档，然后用 Reranker 精排。虽然增加了 100ms 延迟，但准确率提升了 7%，这个 trade-off 是值得的。"

---

### 亮点 2: 流式输出 (User Experience)

**问题**: 大模型生成等待时间长，用户体验差

**解决方案**:
```python
# SSE 流式输出
async def stream_chat():
    async for chunk in llm.astream(prompt):
        yield f"data: {json.dumps({'chunk': chunk})}\n\n"
    yield "data: {\"done\": true}\n\n"
```

**效果**:
- 首字延迟：<500ms
- 用户感知等待时间：减少 60%

**面试话术**:
> "为了提升用户体验，我实现了流式输出。使用 SSE (Server-Sent Events) 技术，LLM 每生成一个 token 就推送给前端。这样用户几乎立即看到响应，感知等待时间减少了 60%。实现上需要注意缓冲和错误处理。"

---

### 亮点 3: 树状会话设计 (Innovation)

**问题**: 长对话上下文管理困难，无法回溯

**解决方案**:
```
会话树结构:
main_session
├── branch_1 (修复 bug)
│   └── 回溯到 main
├── branch_2 (实验功能)
└── branch_3 (代码审查)
```

**优势**:
- 支持 side-quest 不污染主上下文
- 可回溯到任意节点
- 自动总结分支内容

**面试话术**:
> "我设计了树状会话系统。当需要临时处理其他任务时，可以创建分支，完成后回溯到主干。这样既保持了主上下文的纯净，又不丢失分支的工作。这个设计灵感来自 Git 的分支概念。"

### RAG 流程

```
用户问题
    ↓
1. 问题嵌入 (Embedding)
    ↓
2. 向量检索 (Milvus top-50)
    ↓
3. 重排序 (Reranker top-5)
    ↓
4. 构建 Prompt (检索结果 + 历史)
    ↓
5. LLM 生成 (Qwen3-VL)
    ↓
6. 返回答案 + 来源
```

**优化点**:
- 缓存层：相似 query 直接返回
- 异步处理：检索和生成并行
- 超时控制：防止 LLM 卡住

## ❓ 常见问题与答案

### RAG 相关问题

#### Q1: RAG 相比微调的优势是什么？

**A**:
1. **知识更新快**: RAG 只需更新知识库，微调需要重新训练
2. **可解释性强**: 可以追溯答案来源
3. **成本低**: 不需要大量标注数据
4. **幻觉少**: 基于检索到的事实生成

**话术**:
> "RAG 的最大优势是知识更新成本低。比如我们项目，新增文档只需插入向量库，而微调需要重新训练。而且 RAG 可以追溯来源，更容易建立信任。"

---

#### Q2: 如何提升 RAG 的检索准确率？

**A**:
1. **文档切分优化**: 按语义切分，保持完整性
2. **重排序**: 召回后精排
3. **Query 改写**: 扩展同义词、纠正拼写
4. **混合检索**: BM25 + 向量
5. **元数据过滤**: 缩小检索范围

**话术**:
> "我主要做了三点优化：首先是文档切分，按标题层级保持语义完整；其次是重排序，召回 50 篇后精排 5 篇；最后是元数据过滤，可以按文件名、日期等缩小范围。这三项让准确率从 75% 提升到 85%。"

---

#### Q3: 如何处理长文档？

**A**:
1. **智能切分**: 按章节/段落切分
2. **重叠切分**: 保持上下文连续
3. **层次化索引**: 文档级 + 段落级
4. **摘要索引**: 先检索摘要再检索正文

**话术**:
> "对于长文档，我采用层次化切分。首先按章节切分成大块，然后在大块内按段落切分。检索时先定位章节，再精确定位段落。这样既保证了检索速度，又不丢失细节。"

---

### Milvus 相关问题

#### Q4: Milvus 的相似度度量有哪些？

**A**:
- **COSINE**: 余弦相似度 (推荐，最常用)
- **L2**: 欧氏距离
- **IP**: 内积 (等同于 COSINE 对于归一化向量)

**选择建议**:
- 文本嵌入：COSINE
- 图像嵌入：COSINE 或 L2
- 归一化向量：IP (更快)

---

#### Q5: 如何优化 Milvus 的检索性能？

**A**:
1. **索引选择**: HNSW > IVF_FLAT > FLAT
2. **参数调优**: M, efConstruction, efSearch
3. **缓存**: 热点 query 缓存
4. **分区**: 按类别/时间分区
5. **量化**: 标量量化减少内存

**话术**:
> "我主要做了三点优化：第一是使用 HNSW 索引，M=16, efConstruction=200；第二是添加 Redis 缓存，相似 query 直接返回；第三是按知识库分区，减少单次检索范围。优化后从 500ms 降到 50ms。"

## 🔥 项目难点与解决方案

### 难点 1: Milvus etcd WAL 日志疯涨

**问题描述**:
- etcd 的 WAL 日志快速增长
- 几天内占用数十 GB 磁盘
- 导致磁盘空间不足

**根本原因**:
- etcd 默认配置不适合高写入场景
- WAL 日志清理不及时

**解决方案**:
```yaml
# etcd 配置优化
auto-compaction-mode: periodic
auto-compaction-retention: "1h"
quota-backend-bytes: 8589934592  # 8GB
```

**运维措施**:
```bash
# 定期清理脚本
0 2 * * * /path/to/cleanup_etcd.sh
```

**面试话术**:
> "这是部署中遇到的最大坑。etcd 默认配置不适合我们的写入频率。我做了两件事：第一是优化 etcd 配置，开启自动压缩；第二是添加监控，磁盘占用超过 80% 时告警。之后问题就解决了。"

---

### 难点 2: 大 PDF 文件上传超时

**问题描述**:
- >50MB 的 PDF 上传经常超时
- Nginx 默认限制 1MB
- 解析时间长，请求超时

**解决方案**:
```python
# 1. Nginx 配置
client_max_body_size 100M;

# 2. 分片上传
async def upload_chunk(file_id, chunk, index):
    ...

# 3. 后台处理
@app.post("/upload")
async def upload(file: UploadFile):
    task_id = await queue.add(parse_pdf, file)
    return {"task_id": task_id}  # 立即返回
    
@app.get("/status/{task_id}")
async def status(task_id):
    return await queue.get_status(task_id)
```

**面试话术**:
> "大文件上传有三个问题：Nginx 限制、传输超时、解析超时。我的方案是：第一调整 Nginx 配置到 100MB；第二实现分片上传，支持断点续传；第三后台异步处理，上传和解析分离。这样用户体验好很多。"

## 📊 STAR 法则案例

### 案例 1: 性能优化

**Situation (情境)**:
> "项目上线后，用户反馈检索速度慢，平均响应时间 2 秒，高峰期达到 5 秒。"

**Task (任务)**:
> "我的目标是将响应时间优化到 500ms 以内，提升用户体验。"

**Action (行动)**:
> "我做了性能分析，发现瓶颈在三个地方：
> 1. Milvus 检索慢：使用 FLAT 索引，全量扫描
> 2. 重复计算：相同 query 重复嵌入和检索
> 3. 同步阻塞：检索和生成串行
>
> 我的优化方案：
> 1. 将索引改为 HNSW，M=16, efConstruction=200
> 2. 添加 Redis 缓存，缓存相似 query 的结果
> 3. 异步化，检索和 LLM 调用并行"

**Result (结果)**:
> "优化后效果显著：
> - 平均响应时间：2000ms → 400ms (80% 提升)
> - P99 延迟：5000ms → 800ms
> - 缓存命中率：65%
> - 用户满意度：75% → 92%
>
> 这次优化让我深刻理解了性能分析的重要性，不能盲目优化，要先找到瓶颈。"

## 💻 代码示例

### RAG 核心代码

```python
class RAGService:
    def __init__(self):
        self.milvus = MilvusClient()
        self.llm = AsyncOpenAI(...)
        self.reranker = Reranker()
    
    async def chat(self, query: str, collection: str) -> str:
        # 1. 生成嵌入
        embedding = await self.embed(query)
        
        # 2. 向量检索
        results = await self.milvus.search(
            collection=collection,
            vector=embedding,
            top_k=50
        )
        
        # 3. 重排序
        ranked = await self.reranker.rerank(
            query=query,
            documents=results,
            top_n=5
        )
        
        # 4. 构建 Prompt
        prompt = self.build_prompt(query, ranked)
        
        # 5. LLM 生成
        response = await self.llm.chat.completions.create(
            model="qwen3-vl-plus",
            messages=[{"role": "user", "content": prompt}],
            stream=True
        )
        
        return response
```
