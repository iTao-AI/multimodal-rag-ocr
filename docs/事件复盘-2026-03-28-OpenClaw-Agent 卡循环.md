## 🔬 根因分析

### 直接原因

**Sebastian 陷入 Tool 调用无限循环**

```
触发点：用户说"Web 端显示的还是 3.23 版本"
    ↓
Sebastian 判断：需要刷新缓存/重启 gateway
    ↓
执行：openclaw gateway restart
    ↓
问题：Tool 执行超时/无响应/返回格式错误
    ↓
AI 重试逻辑：没有次数上限
    ↓
结果：每分钟 2 次 Tool 调用 × 20 分钟 = 40+ 次调用
    ↓
后果：3 个 agent 实例同时卡住，CPU 100% 空转
```

### 深层原因

| 问题层级 | 原因 | 影响 |
|---------|------|------|
| **代码层** | Tool 调用无重试上限 | AI 可以无限重试 |
| **架构层** | 会话隔离失效 | 一个会话卡住，新会话继续创建 |
| **运维层** | 无健康检查告警 | 无法及时发现 CPU 异常 |
| **认知层** | 浏览器缓存误判为服务端问题 | 采取了错误的修复方案 |

## 💡 经验教训

### 技术层面

1. **Tool 调用必须有上限** - 次数限制 + 超时限制
2. **浏览器缓存问题不需要重启服务端** - 先问用户是否清缓存
3. **多进程架构需要健康检查** - 自动检测 + 自动恢复
4. **会话隔离很重要** - 防止一个会话卡住影响全局

### 运维层面

1. **监控告警不可少** - CPU、内存、会话数
2. **快速诊断脚本** - 一键检查 + 一键修复
3. **文档化常见问题** - 减少重复排查时间

### 沟通层面

1. **用户说"Web 显示旧版本" = 清缓存** - 不是重启
2. **先问清楚再行动** - 避免误判
3. **解释原因** - 让用户理解问题本质

## 🎓 学习要点

### 对于 AI Agent 开发者

1. **Tool 调用设计**
   - 必须有重试上限
   - 必须有超时机制
   - 必须有错误处理

2. **会话管理**
   - 会话隔离
   - 资源限制（CPU、内存）
   - 自动清理超时会话

3. **健康检查**
   - 进程级监控
   - 会话级监控
   - 自动恢复机制

### 对于运维人员

1. **快速诊断流程**
   ```bash
   # 1. 检查进程
   ps aux | grep openclaw
   
   # 2. 检查会话
   openclaw sessions list
   
   # 3. 卡住了就 kill + restart
   kill -9 <pid> && openclaw gateway restart
   ```

2. **常见误区**
   - Web UI 版本问题 ≠ 服务端问题
   - 重启 ≠ 万能解决方案
   - 多进程 ≠ 多实例（是正常设计）

### 对于用户

1. **浏览器缓存问题**
   - 硬刷新：Cmd+Shift+R
   - 清缓存：浏览器设置
   - 无痕模式：排除缓存干扰

2. **报告问题的正确姿势**
   - 截图（包含版本号）
   - 说明操作步骤
   - 提供错误信息
