### 版本 3：STAR 法则版（适合面试）

**情境（Situation）**：
在使用 OpenClaw AI Agent 系统时，遇到 AI Agent 陷入 Tool 调用无限循环，导致服务中断 26 分钟，飞书消息无法回复。

**任务（Task）**：
需要快速诊断问题并恢复服务，同时分析根因防止再次发生。这是一个 P2 级生产事故，影响用户正常使用 AI 管家服务。

**行动（Action）**：
1. **快速诊断**：通过 `ps aux | grep openclaw` 发现 3 个 CPU 100% 的异常进程
2. **日志分析**：查看 Tool 调用日志，发现 20 分钟内调用 40+ 次（正常应为 0-2 次）
3. **根因定位**：
   - 直接原因：`openclaw gateway restart` Tool 执行超时，AI 重试逻辑无上限
   - 深层原因：会话隔离失效、无健康检查、浏览器缓存误判
4. **紧急修复**：`kill -9` 终止卡住的进程 → 重启 gateway → 验证状态
5. **长期优化**：产出 5423 字复盘文档，提出 5 项改进建议

**结果（Result）**：
- 服务在 26 分钟内恢复，用户感知中断时间 <30 分钟
- 产出标准化诊断流程和健康检查脚本
- 复盘文档可作为团队 SOP 和新人培训资料
- 个人能力提升：AI Agent 系统理解、故障诊断、系统化分析

**诊断过程**：
1. 我先查看活动监视器，发现 3 个 OpenClaw 进程 CPU 100% 空转
2. 然后查看 Tool 调用日志，发现 20 分钟内调用了 40+ 次（正常应该是 0-2 次）
3. 进一步分析发现是 `gateway restart` Tool 执行超时，AI 的重试逻辑没有上限，导致无限循环

**解决方案**：
1. 紧急修复：`kill -9` 终止卡住的进程，重启 gateway，26 分钟恢复服务
2. 长期优化：产出 5423 字复盘文档，提出 Tool 调用上限、健康检查等 5 项改进建议

**经验教训**：
1. AI Tool 调用必须有重试上限（次数 + 超时）
2. 多进程架构需要健康检查和自动恢复机制
3. 浏览器缓存问题不需要重启服务端（这是误判的根因）

这个经历让我深刻理解了 AI Agent 系统的脆弱性和运维的重要性。"

### 3. "AI Agent 理解"类问题

**面试官**："你如何理解 AI Agent 系统？"

**回答**：

"基于我的实战经验，AI Agent 系统有三个关键层面：

**1. 架构层面**：
- 多进程设计（Gateway + 多 Agent 实例）是必要的，可以实现会话隔离
- 但需要防止一个会话卡住影响全局，需要资源限制和自动 kill 机制

**2. Tool 调用层面**：
- Tool 是 Agent 与外部世界交互的接口，但必须有重试上限和超时机制
- 我遇到的事故就是 Tool 调用无上限导致的无限循环

**3. 运维层面**：
- AI Agent 不是部署完就结束了，需要健康检查、监控告警、自动恢复
- 我提出了 5 项改进建议，包括 CPU 监控、watch dog、告警通知等

**总结**：AI Agent 系统 = AI 能力 × 工程化 × 运维保障，三者缺一不可。"

### 4. "失败经历"类问题

**面试官**："分享一次失败的经历，你学到了什么？"

**回答**：

"我处理 OpenClaw 事故时，一开始误判了问题。

**失败**：用户说'Web 端显示旧版本'，我第一反应是重启 gateway，结果导致 Tool 调用无限循环，服务中断 26 分钟。

**反思**：
1. 没有先问清楚问题本质（其实是浏览器缓存）
2. 没有评估操作风险（重启 gateway 可能导致会话丢失）
3. 没有设置安全网（Tool 调用无上限）

**改进**：
1. 现在我会先问清楚用户的具体操作和现象
2. 提供多个解决方案（清缓存、硬刷新、无痕模式）
3. 任何自动化操作都要有重试上限和超时机制

这个失败让我学会了：**先诊断，再行动；先问清楚，再动手**。"
