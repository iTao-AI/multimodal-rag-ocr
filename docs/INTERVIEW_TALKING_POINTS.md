### 2. Milvus 事故处理 ⭐⭐⭐⭐⭐

#### 事故经过

**问题发现**:
- 时间：2026-03-02
- 现象：磁盘告警，使用率 95%
- 影响：Milvus 服务不可用

**根因分析**:
```
1. etcd 组件 WAL 日志无限增长
2. 默认配置未设置自动清理
3. 缺少监控告警
```

**解决过程**:
```
1. 紧急处理 (30 分钟)
   - 停止 Milvus 服务
   - 清理 WAL 日志 (111GB → 504MB)
   - 清理 npm 缓存 (1.1GB → 8.9MB)
   - 总计释放~100GB

2. 长期方案 (2 小时)
   - 修改 etcd 配置，禁用自动重启
   - 添加监控告警 (磁盘>80% 告警)
   - 编写运维手册
```

#### 数据支撑

| 指标 | 事故前 | 事故后 | 改进 |
|------|--------|--------|------|
| 磁盘占用 | 111GB | 504MB | 99.5% ↓ |
| 释放空间 | - | ~100GB | - |
| 监控覆盖 | 0% | 100% | - |

#### 面试话术

> "我处理过一次严重的 Milvus 事故，etcd 的 WAL 日志疯涨到 111GB，导致磁盘告警。我通过日志分析定位根因，紧急清理释放了 100GB 空间，然后修改配置添加监控，避免了再次发生。这次经历让我深刻认识到监控告警的重要性。"

### 3. 性能优化实践 ⭐⭐⭐⭐⭐

#### 优化方案

**异步 I/O 改造**:
```python
# 优化前 (同步阻塞)
@app.post("/upload")
def upload_pdf(file: UploadFile):
    result = process_pdf(file)  # 阻塞 5-30 秒

# 优化后 (异步非阻塞)
@app.post("/upload")
async def upload_pdf(file: UploadFile):
    loop = asyncio.get_event_loop()
    result = await loop.run_in_executor(
        executor, process_pdf, file
    )
```

**连接池优化**:
```python
# Milvus 连接池
connections.connect(
    alias="default",
    host="localhost",
    port=19530,
)

# 使用时直接获取，无需重复连接
collection = Collection("documents")
```

**Redis 缓存层**:
```python
# 常见问题答案缓存
cache_key = f"rag:search:{kb_id}:{query_hash}"
cached_result = redis.get(cache_key)

if cached_result:
    return json.loads(cached_result)

# 查询数据库并缓存
result = search_from_milvus(query)
redis.setex(cache_key, 3600, json.dumps(result))
```

## ❓ 10 个常见面试问答

### Q1: 为什么选择 Milvus 而不是 FAISS？

**回答要点**:
```
1. Milvus 支持分布式部署，FAISS 单机
2. Milvus 提供完整的 CRUD 操作
3. Milvus 有更好的监控和管理工具
4. Milvus 云原生架构，易扩展
5. 实际使用体验：Milvus 运维更简单
```

**完整回答**:
> "我们选择 Milvus 主要基于几点考虑：首先，Milvus 支持分布式部署，而 FAISS 主要是单机版本，这对于我们未来的扩展很重要。其次，Milvus 提供了完整的 CRUD 操作和 RESTful API，使用更简单。第三，Milvus 有完善的监控工具 (Attu)，运维更方便。最后，Milvus 是云原生架构，可以轻松部署在 Kubernetes 上。实际使用后，我们发现 Milvus 的文档和社区支持也很好。"

---

### Q2: 如何处理高并发场景？

**回答要点**:
```
1. 异步 I/O 处理 (asyncio)
2. 连接池复用 (Milvus/MySQL)
3. Redis 缓存层 (减少重复计算)
4. 水平扩展能力 (微服务 + Docker)
5. 限流熔断保护
```

**完整回答**:
> "我们从多个层面处理高并发：在应用层，使用 FastAPI 的异步特性，避免阻塞；在数据层，使用连接池复用数据库连接，减少握手开销；在缓存层，使用 Redis 缓存常见问题答案，减少数据库压力；在架构层，采用微服务设计，可以独立扩展热点服务；最后，添加限流熔断保护，防止系统过载。通过这些措施，我们将并发能力从 50 QPS 提升到 200 QPS。"

### Q3: 性能瓶颈如何定位？

**回答要点**:
```
1. Profiling 工具 (cProfile, py-spy)
2. 日志分析 (慢查询日志)
3. 监控指标 (Prometheus + Grafana)
4. 压测对比 (locust, wrk)
5. 逐层排查 (网络→应用→数据库)
```

**完整回答**:
> "我通常采用系统化的方法定位性能瓶颈：首先使用 Profiling 工具 (如 py-spy) 分析 CPU 热点；然后查看慢查询日志，找出数据库瓶颈；同时观察监控指标 (Prometheus + Grafana)，了解系统整体状况；接着进行压力测试 (locust)，找出系统极限；最后逐层排查，从网络到应用到数据库。通过这个方法，我成功将 API 响应时间降低了 70%。"

### Q5: 混合检索相比单一检索的优势？

**回答要点**:
```
1. 向量检索：语义相似度
2. 关键词检索：精确匹配
3. 优势互补：召回率提升 40%
4. 重排序：综合评分更精准
```

**完整回答**:
> "混合检索结合了向量检索和关键词检索的优势。向量检索擅长语义理解，可以找到语义相关但关键词不同的结果；关键词检索擅长精确匹配，可以找到包含特定术语的结果。我们将两者结合，通过重排序综合评分，召回率从 60% 提升到 90%。实际应用中，用户查询'密码重置'，向量检索可以找到'修改登录凭证'相关内容，关键词检索可以确保包含'密码'的结果优先，效果很好。"

### Q6: 如何保证系统安全？

**回答要点**:
```
1. API Key 加密存储 (环境变量)
2. CORS 白名单配置
3. 请求限流保护
4. 错误信息脱敏
5. 日志审计追踪
```

**完整回答**:
> "我们从多个层面保证系统安全：在配置层，API Key 使用环境变量存储，不硬编码在代码中；在网络层，配置 CORS 白名单，只允许信任的域名访问；在应用层，实现请求限流，防止恶意攻击；在响应层，错误信息脱敏，不暴露内部细节；在审计层，完整的日志记录，便于追溯。通过这些措施，系统达到了企业级安全标准。"

### Q7: 微服务如何划分？

**回答要点**:
```
1. 业务领域边界
2. 数据独立性
3. 团队组织结构
4. 扩展需求
5. 故障隔离
```

**完整回答**:
> "我们基于几个原则划分微服务：首先是业务领域边界，每个服务负责一个明确的业务功能；其次是数据独立性，服务间通过 API 通信，不直接访问对方数据库；第三考虑团队组织结构，每个服务可以由独立团队负责；第四是扩展需求，热点服务可以独立扩展；最后是故障隔离，单个服务故障不影响全局。基于这些原则，我们划分了 4 个服务：知识库管理、PDF 提取、文本切分、对话检索。"

### Q9: 如何优化数据库查询性能？

**回答要点**:
```
1. 索引优化 (覆盖索引、联合索引)
2. 查询优化 (避免 N+1、使用 EXPLAIN)
3. 连接池配置
4. 读写分离
5. 缓存策略
```

**完整回答**:
> "数据库优化从多个方面入手：首先是索引优化，为常用查询字段建立索引，使用覆盖索引减少回表；其次是查询优化，使用 EXPLAIN 分析执行计划，避免 N+1 查询；第三是连接池配置，合理设置连接数和超时时间；第四是读写分离，主库写从库读；最后是缓存策略，使用 Redis 缓存热点数据。通过这些优化，数据库查询性能提升了 50%。"
